# 多租户改造七期说明

第七期开始处理“登录模型租户化”。这一期的目标不是马上把 `username` 改成租户内唯一，而是先把登录、会话和登录页入口改成显式带租户上下文，给后面的真正租户级登录模型留出平滑过渡空间。

## 本期范围

- 登录接口支持 `tenantIdentifier`
  - 可传租户编码 `tenantCode`
  - 也可传租户标识 `tenantSlug`
- 登录页新增“租户”选择入口
- 新增公开租户列表接口，供登录页加载可选租户
- Access Token / Refresh Token 都带租户声明
- 刷新会话时继续沿用当前用户的租户上下文
- 顶栏显示当前登录租户，避免误入别的租户后台

## 后端变更

- `lib/tenant-foundation.js`
  - 新增 `normalizeTenantIdentifier()`
  - 新增 `resolveTenantByIdentifier()`
  - 新增 `listEnabledTenants()`
  - `tenant` 结构补充 `tenantSlug`

- `lib/auth.js`
  - 登录时改成“用户名 + 所选租户”联合校验
  - 未指定租户时，默认走默认租户
  - 访问令牌和刷新令牌新增：
    - `tenantId`
    - `tenantCode`
    - `tenantSlug`
  - `authenticateRequest()` 会校验 token 租户声明与当前用户租户是否一致

- `routes/auth.js`
  - `POST /api/v1/auth/login` 支持 `tenantIdentifier`
  - 新增 `GET /api/v1/auth/tenant-options`

## 前端变更

- 登录页新增租户选择
  - 默认选中默认租户
  - 支持通过 `?tenant=xxx` 预填

- 管理后台页头显示当前租户

## 本期价值

- 登录链路已经正式具备“租户上下文”
- 默认租户现网账号和设备不受影响
- 为后面处理 `username` 是否租户内唯一打好了基础
- 后续如果按租户发独立登录入口或域名，前台已经有承接点

## 本期未覆盖

- `username` 仍然是全局唯一
- 还没有做租户管理后台页面
- 还没有做子域名/专属域名自动识别租户
- 还没有做租户管理员初始化向导

下一期建议：

- 正式评估 `username` 的租户内唯一改造
- 新增租户管理后台
- 把设备接入凭证从“每租户一个默认 token”升级成更细粒度模型
