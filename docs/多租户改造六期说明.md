# 多租户改造六期说明

第六期开始处理“设备与媒体编码租户化”。这一期的目标不是改登录模型，而是先把最容易发生跨租户冲突的设备接入编码切到租户范围，同时让设备侧 token 解析真正落到租户上下文。

## 本期范围

- `iot_gateways.gateway_code` 改成租户内唯一
- `iot_sensors.sensor_code` 改成租户内唯一
- `iot_actuators.actuator_code` 改成租户内唯一
- `iot_media_nodes.node_code` 改成租户内唯一
- `iot_cameras.camera_code` 改成租户内唯一
- 设备上报、设备控制查询、设备控制回报改成先解析 token，再绑定租户
- 设备自动接入创建网关和传感器时，按当前租户查重
- 演示种子脚本中所有按设备/媒体编码查父对象的语句，全部补上默认租户或关联租户条件

## 数据库变更

- 新增迁移脚本：`sql/140_multi_tenant_device_codes.sql`

迁移逻辑分两步：

1. 先回填空租户
   - `iot_gateways`
   - `iot_sensors`
   - `iot_actuators`
   - `iot_media_nodes`
   - `iot_cameras`

2. 再切唯一索引
   - `uk_iot_gateways_gateway_code` -> `uk_iot_gateways_tenant_gateway_code (tenant_id, gateway_code)`
   - `uk_iot_sensors_sensor_code` -> `uk_iot_sensors_tenant_sensor_code (tenant_id, sensor_code)`
   - `uk_iot_actuators_actuator_code` -> `uk_iot_actuators_tenant_actuator_code (tenant_id, actuator_code)`
   - `uk_iot_media_nodes_node_code` -> `uk_iot_media_nodes_tenant_node_code (tenant_id, node_code)`
   - `uk_iot_cameras_camera_code` -> `uk_iot_cameras_tenant_camera_code (tenant_id, camera_code)`

当前仍然保持全局唯一的高风险字段：

- `iot_gateways.serial_no`
- `iot_cameras.serial_no`

## 代码变更

- `routes/iot.js`
  - 从“只校验 token 是否存在”升级为“解析 token 对应的租户”
  - `ingest / device-control / device-control/report` 都会先绑定租户
  - 设备上报限流键补上租户维度，避免不同租户同编码设备互相影响
  - `ensureGateway()` 和 `ensureSensor()` 改成租户内查重

- `sql/010_core_schema.sql`
  - 更新网关、传感器、执行器唯一索引定义

- `sql/060_camera_media_extension.sql`
  - 更新媒体节点、摄像头唯一索引定义

- `sql/030_seed_demo_data.sql`
  - 默认租户演示设备、媒体和抓图数据改成默认租户限定

## 本期价值

- 多个租户可以复用相同的 `gateway_code / sensor_code / actuator_code`
- 多个租户可以复用相同的 `camera_code / node_code`
- 设备 token 不再只是“允许接入”，还会成为“接入到哪个租户”的边界
- 同编码设备在不同租户下不会再共享控制状态或自动接入档案

## 本期未覆盖

- `username` 仍然是全局唯一
- 设备和摄像头的 `serial_no` 仍然是全局唯一
- 还没有做“每个租户独立的设备 token 列表”或“每设备单独 token”
