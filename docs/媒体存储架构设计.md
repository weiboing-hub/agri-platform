# 媒体存储架构设计

## 目标

媒体文件后续统一按“对象存储优先、数据库只存元数据”的方式演进：

- 图片、缩略图、录像文件放对象存储
- MySQL 仅保留文件索引、业务归属和时间信息
- 路径从一开始就按租户维度组织，后续迁移不改主数据模型

## 推荐结构

- 文件本体：腾讯云 COS / 其他对象存储
- 元数据：MySQL
- 访问层：前端通过平台返回的 `fileUrl / thumbnailUrl` 访问

## 路径规范

建议统一对象键前缀：

```text
tenant/{tenantCode}/camera/{cameraCode}/snapshot/{sourceType}/{YYYY}/{MM}/{DD}/{snapshotNo}.jpg
tenant/{tenantCode}/camera/{cameraCode}/snapshot/{sourceType}/{YYYY}/{MM}/{DD}/thumb/{snapshotNo}-thumb.jpg
tenant/{tenantCode}/camera/{cameraCode}/recording/{YYYY}/{MM}/{DD}/{recordingNo}.mp4
```

说明：

- `tenantCode`：租户编码
- `cameraCode`：摄像头编码
- `sourceType`：`manual / schedule / event / ftp-upload`

## 当前实现

本期已完成：

- 新增 `admin-api/src/lib/media-storage.js`
- 手动抓图和 FTP 抓图元数据都改为写入租户化对象键
- 时间轴接口新增：
  - `fileUrl`
  - `thumbnailUrl`
- 系统设置新增 `media_storage` 分组，用于维护：
  - `storage_provider`
  - `public_base_url`
  - `bucket_name`
  - `region`
  - `path_prefix`
  - `local_root_path`
  - `local_public_base_url`

## 当前边界

当前还没有做真实对象存储上传，现阶段属于“路径和配置先统一”：

- 数据库里记录的是标准化对象键
- 如果配置了 `public_base_url`，接口会自动拼出可访问 URL
- 如果仍然使用本地模式，可以先保持 `storage_provider=local`

## 后续接 COS 的最小改造

后面接腾讯云 COS 时，只需要继续补这三步：

1. 在后端新增 COS 上传适配器
2. 抓图文件写入 COS 后，把对象键写入 `file_path / thumbnail_path`
3. 在系统设置里填入 `public_base_url / bucket_name / region`

这样前端和数据库都不需要重构。
