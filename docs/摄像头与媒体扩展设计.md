# 摄像头与媒体扩展设计

## 1. 目标

当前平台已经具备传感器、告警、控制、AI 报告和设备管理能力，但还不具备完整的视频接入与图像证据链。  
本扩展的目标是把以下能力补齐：

- WiFi 摄像头接入
- RTSP 拉流
- RTMP/HLS/WebRTC 转发
- FTP 图片/录像接收
- 定时抓图与告警联动抓图
- 图片中心与录像中心
- 图像 AI 分析结果

## 2. 当前缺口

当前项目里虽然菜单设计上已经预留了“摄像头管理 / 历史图片 / 图像分析结果”，但还缺少以下基础能力：

- 摄像头主数据模型
- 视频流模型
- 媒体服务节点模型
- 抓图任务与抓图结果模型
- 录像索引模型
- 图像分析结果模型
- 媒体服务层
- FTP 文件接收机制
- RTSP 到浏览器可播放协议的转码链路

## 3. 总体架构建议

```text
摄像头(WiFi/RTSP/FTP)
        ↓
media-service
  - 拉 RTSP
  - 转 HLS/FLV/WebRTC
  - 定时抓图
  - 接收 FTP 上传
  - 管理录像文件
        ↓
admin-api
  - 摄像头配置
  - 抓图任务调度
  - 录像/图片元数据
  - 告警联动
  - AI 图像分析调度
        ↓
MySQL + 对象存储/磁盘
```

核心原则：

- `admin-api` 只管理元数据和业务逻辑，不直接承担重媒体转码
- `media-service` 负责流媒体与文件处理
- MySQL 只存索引和业务状态，不存图片二进制
- 图片、录像建议落本地磁盘、NAS、MinIO 或 OSS

## 4. 协议分工建议

### WiFi

- 仅表示摄像头联网方式
- 不承担媒体播放协议职责

### RTSP

- 作为摄像头原始拉流协议
- 用于媒体服务接入源流

### RTMP

- 用作平台内部转推或向外部平台推流
- 不建议直接作为浏览器主播放协议

### FTP

- 用于摄像头定时上传 JPEG 图片或录像文件
- 适合低端设备和“定时抓图”型摄像头

### 浏览器播放协议

- 建议实际输出为 `HLS / WebRTC / FLV`
- 前端不要直接播放 RTSP

## 8. 关键联动建议

### 告警联动

- 低湿、高温、异常规则触发时自动抓图
- 关键告警自动保留前后证据图片
- 告警详情展示关联图片

### AI 联动

- AI 诊断时可附带最近一张图片
- AI 日报可插入重点时段图片
- 图像分析结果可回写到 AI 报告摘要

### 控制联动

- 手动控制前可查看当前画面
- 控制后自动抓图留档

## 11. 建议的最小可行落地

如果你希望先快速上线一个有价值的版本，建议只做这四项：

- 摄像头管理
- 手动抓图
- 告警联动抓图
- 历史图片时间轴

这样复杂度最低，但已经能形成“告警 + 图片证据 + AI 诊断素材”的闭环。
